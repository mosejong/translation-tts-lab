## 발표에서 강조한 구현

SchoolBridge의 번역 파트는 "NLLB로 번역했다"에서 멈추지 않고, 학교 안내문에서 실제 행동 오류로 이어질 수 있는 값을 보호하는 구조로 설계했다.

![SchoolBridge 전체 파이프라인](assets/schoolbridge-overall-pipeline.png)

![NLLB 핵심 정보 보존 파이프라인](assets/nllb-slot-protection-pipeline.png)

| 구성 | 역할 |
| --- | --- |
| Slot Protection | 날짜, 금액, URL, 전화번호를 `__SLOT0__` 형태로 격리한 뒤 번역 후 복원 |
| Glossary Injection | 학교 도메인 용어를 권장 번역어로 고정해 NLLB 오역을 줄임 |
| Template-based translation | 준비물, 제출물, 납부 등 반복 문장은 구조화 후 템플릿으로 번역 |
| Unknown-term loop | glossary에 없는 용어를 수집해 다음 번역 품질 개선 후보로 남김 |
| TTS 연결 | 번역 결과를 베트남어 음성 안내까지 이어지게 구성 |

## 받은 피드백

좋았던 점:

- 슬롯, 마스킹, glossary injection, template 기반 번역 구조가 실제 서비스 문제를 잘 잡았다.
- 단순 모델 호출이 아니라 NLLB의 약점을 알고 보완한 구현으로 보였다.
- 구현 결과를 수치로 설명해 설득력이 있었다.
- 6팀 중 문제 정의와 페르소나 완성도가 가장 높았다.
- 베트남 출신 30대, 한국어 읽기 3.82/5, 가정통신문 어휘 37% 미등재까지 구체화한 페르소나가 기술 선택과 기능 우선순위의 근거가 되었다.
- 3,300장 직접 수집, 6만 행 라벨링, 자체 모델 2개 학습까지 이어지는 데이터 파이프라인을 팀이 역할 분담해 완수했다.
- 3단계 자체화 시도와 실패 과정을 솔직히 남긴 엔지니어링 태도, 팀 전체 협업이 높게 평가되었다.
- 문제를 데이터로 증명하고, 팀이 함께 데이터를 만들고, 그 데이터로 모델을 개선하는 사이클을 3주 안에 완성한 점이 실제 AI 프로덕트 개발 방식에 가까웠다는 평가를 받았다.

아쉬운 점:

- 학습과 출력 처리 흐름을 한 장 더 자세히 보여주면 더 좋다.
- 9개 언어 지원을 핵심 차별점으로 내세웠지만, 성능 평가는 한국어 기준 F1/Recall에 더 집중되었다.
- 언어별 번역 품질 지표가 있었다면 "실제로 9개 언어에서 잘 작동한다"는 주장의 설득력이 더 높아졌을 것이다.
- 데모 시연에서 화면은 보여줬지만 구두 설명이 부족해 청중이 기능을 충분히 이해하지 못한 점이 아쉬웠다.
